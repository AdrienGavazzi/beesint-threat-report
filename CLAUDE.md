# CLAUDE.md

Guidance for Claude Code working in `beesint-threat-report` — the weekly cyber threat intel ETL pipeline. Separate repo from `beesint-backend`/`beesint-frontend`/`beesint-jobs`; see root `CLAUDE.md`'s "Threat Report Pipeline" section for how it fits into the rest of BeeSINT.

---

## Architecture

Cron-driven (GitHub Actions, see `.github/workflows/cron-etl.yml`), no server process. Entry point: `orchestrate.py::run()`.

Layers, in order:
1. **`extract/`** — one module per source (`nvd.py`, `kev.py`, `feodo.py`, `urlhaus.py`, `threatfox.py`), each an async `httpx` call returning a raw `polars` DataFrame. Every extractor degrades independently: a failed source returns an empty frame + a `sources_status["<name>"] = "failed"` entry, the run continues rather than aborting (never let one dead API kill the whole report).
2. **`transform/`** — `dedup.py`, `diffing.py` (new-vs-seen-before), `geoloc.py` (ip-api.com batch), `kpis.py` (`ReportKpis`), `mttk.py` (mean-time-to-KEV), `ranking.py` (`rank_top_n_cves`/`rank_top_n_ips`/`rank_top_n_urls` — every list in the report is top-N bounded here, keeps PDF size/page-count sane).
3. **`validate/`** — `schemas.py` (Pydantic row models), `frames.py` (pandera frame-level schemas).
4. **`load/`** — `pdf_context.py` (the single place raw data becomes template-ready dicts: chart SVGs, chip-list aggregations, date formatting, lineage), `pdf_renderer.py` (Jinja env + WeasyPrint call), `json_writer.py` (public JSON API payload — shares the same item-level dicts as the PDF context, so a data fix in `orchestrate.py`'s `_build_top_cves`/`_build_top_ips` affects both surfaces at once).

Output: PDF (WeasyPrint) + JSON (public API) + entry appended to S3 `runs/index.json`, then a webhook POST to the backend (`publish/webhook.py`) unless `dry_run` (no `BACKEND_WEBHOOK_URL` configured).

## Data source integration rule

New sources must **enrich an existing tracked entity**, never become their own isolated report section. A source describing a C2 IP (Shodan InternetDB, Spamhaus DROP/EDROP, GreyNoise) merges into the existing `c2_items` record for that IP. A source describing a malicious URL (OpenPhish) merges into `malicious_url_items` by normalized URL, deduped against URLhaus — never list the same URL twice because two feeds reported it. The payoff of merging is a cross-source signal ("confirmed by N feeds") that's more useful than either source shown alone.

**PhishTank → OpenPhish:** PhishTank was removed (`extract/phishtank.py`, `transform/phishtank_merge.py`) because its signups are closed — no API key is obtainable anymore, so the source could never activate. Replaced by OpenPhish (`extract/openphish.py`, `transform/openphish_merge.py`): a free public feed (`feed.txt`, one URL per line), no API key, no hard rate limit. Same merge role, same theme (malicious URLs), just a different upstream. If PhishTank ever becomes viable again, don't resurrect the old files — OpenPhish already fills this role cleanly.

The merge step runs **before** `transform/ranking.py`'s top-N cut, and cross-confirmed entries should sort ahead of single-source ones within that cut.

**Exception: Ransomware Watch.** `extract/ransomware_live.py` + `transform/ransomware.py` are a deliberate exception to the rule above — ransomware.live describes ransomware groups/leak-site posts, an entity this pipeline had no existing tracked record for (unlike a C2 IP or a malicious URL), so it renders as its own PDF/JSON section instead of enriching something else. Don't "fix" this into a merge later without re-checking that no existing entity actually fits. EPSS (`extract/epss.py`) is the opposite case and follows the normal rule: it enriches existing CVE/KEV records (`epss_score`/`epss_percentile` merged onto `top_cves`/KEV rows), never its own section.

Any new technical field (open ports, CIDR cross-confirmation, IP classification) should get its headline finding echoed as a plain-language sentence in `pdf_context.py::_build_executive_summary()` — see "Target audience" below.

## Target audience

`_build_executive_summary()`'s docstring says "non-expert-cyber recruiter" — that governs only the executive-summary paragraph's *tone* (plain language, no jargon, no "vs last week" on a cold start). Per the actual CDC (`beesint-infra/docs/threat-report.md` §1-2), this component is a **Data Engineer/Cyber portfolio piece** — but the portfolio narrative belongs in the ETL repo's own README (§25), never in the report's own content (confirmed decision: no pipeline-architecture diagram or tech-stack chip strip inside the report itself). The report stays purely technical/functional; the one place worth extra care because a technical reviewer reads it closely is the Pipeline Lineage/Attribution section (run reproducibility, source attribution) — not because it should self-promote, but because it's the section most likely to be scrutinized as engineering work.

## WeasyPrint 62.3 — empirically verified limits

Pinned in `requirements.txt` (`weasyprint==62.3` — don't bump without re-verifying these, `pydyf>=0.12` is known to break the `Stream.transform` API this version relies on). All of the following are documented inline in `styles/report.css` — treat as proven, don't re-test:

- No CSS Grid `repeat(auto-fit, ...)` — silently ignored with a render warning. Use `display: flex; flex-wrap: wrap; gap` instead (already the pattern for `.kpi-grid`, `.chart-row`).
- Nested `flex + gap` inside a `flex-wrap` parent breaks text wrapping for short adjacent tokens (verified on badges). Use `display: inline-block` + `margin` instead for anything wrapping short text (`.chip`, `.source-badge`).
- `<th>`, `<strong>`, `.section-title` need an explicit `font-weight` reset. The UA-stylesheet default bold (700) isn't in any embedded webfont (Syne/Plus Jakarta Sans/JetBrains Mono ship 400/500/600 only) — an unset weight silently falls back to a forbidden system font.
- `@page` margin boxes (`@bottom-left`/`@bottom-center`/`@bottom-right`, `@page :first`) are WeasyPrint/CSS-Paged-Media syntax, not standard browser CSS — don't expect them to behave like anything testable in a normal browser devtools.
- The `@page` background must be set on `@page` itself, not just `body` — `body { background }` only covers the content box, not the margin area (confirmed by the explicit fallback-to-white-frame bug this codebase already hit once).
- No live network calls during PDF render — the world map is a bundled static asset, not a live tile fetch. Deliberate reliability principle: an ETL run must never depend on a third-party tile service being up just to draw a chart. Any new chart-building code in `pdf_context.py` must stay offline too.
- Charts render nothing (`None`) below a real-signal threshold rather than showing a meaningless chart — see `_HISTOGRAM_MIN_ITEMS`, `_build_severity_donut_svg`'s `total > 0` check, and the chip-list-over-bar-chart choice for top countries/vendors (most real counts were 1, a bar chart of all-1s communicates nothing a number doesn't already say). New charts must follow the same discipline.
- **`flex-wrap` + `gap` silently drops the gap once wrapping actually forces multiple lines** (confirmed empirically via a real render, not just CSS reading — distinct from the nested-flex-text-wrap issue above). `.kpi-grid`/`.chart-row` use `flex+gap` safely only because they never have enough cards to wrap past 1 row. `.lineage-grid` had 7 cards wrapping to 3 rows and rendered with literally zero gap (borders touching) until fixed by giving `.lineage-card` an explicit `margin` instead of relying on the parent's `gap` — this class was later removed entirely in the Phase B lineage redesign (sober grouped list, `.lineage-list`/`.lineage-row`, no more card look), but the underlying `flex-wrap`+`gap` finding still stands for any future grid. Any new grid that might wrap to 2+ rows should use the margin pattern from the start, not `gap`.
- **A `flex` item with a fixed `flex: 0 0 Npx` basis and text wider than that basis does not wrap/clip to the basis the way a browser would — it overflows to its natural content width, which visually "eats" the parent's `gap` even on a plain `nowrap` flex row (no wrapping involved at all).** Confirmed empirically on `.lineage-row` (Phase B): 3 fixed/auto-basis children (`flex: 0 0 200px` name, `flex: 1 1 auto` url, `flex: 0 0 200px` note) with `gap: 10px` rendered with the 3 columns flush against each other, zero visible gap, whenever the name text exceeded 200px — distinct from both `flex-wrap`+`gap` (above) and the nested-flex-text-wrap issue, since this row never wraps at all. Fix: drop `gap` in favor of explicit `margin-right` on the fixed-basis children (same margin-over-gap principle as the wrap case), plus `white-space: nowrap; overflow: hidden; text-overflow: ellipsis` on those children so overflow truncates instead of visually devouring the neighboring column's spacing.
- **`content: url(image.png) " text"` inside an `@bottom-left`/`@bottom-right`/`@bottom-center` margin box renders the image, but at its native pixel resolution with no way to constrain it** — a margin box doesn't support a nested selector to style the generated "content" node (no `@bottom-left img { width: ... }` equivalent in CSS Paged Media), and `width`/`height` set on the margin box itself has no effect on the image it generates. Confirmed empirically: a 150×146px logo rendered at that native size inside a ~2.6cm-tall footer margin box completely covered the page. No known workaround on WeasyPrint 62.3 — fall back to text-only footer content when a logo is wanted there (`report.css` `@bottom-left` uses `"BeeSINT — beesint.com"` text for this reason, not the logo asset used elsewhere in the report).
- **SVG root elements with explicit `width`/`height` attributes ignore a CSS rule trying to override them to `100%`** (this was already known for `.map-frame svg`; confirmed to also apply to any other full-width chart card). `_build_history_line_chart_svg`/`_build_area_chart_svg` (`pdf_context.py`) used to set `width="{width}" height="{height}"` pixel attributes on the `<svg>` root — even with a `.chart-card svg.history-line-chart { width: 100% }` CSS rule in place, the chart rendered at its fixed intrinsic pixel width (visibly stopping partway across a full-width card) rather than stretching. Fix is the same as the map: drop the `width`/`height` attributes from the `<svg>` root entirely (keep only `viewBox`), and let the CSS rule do the sizing. `_build_mini_bar_chart_svg`/`_build_stacked_bar_svg`/`_build_cvss_histogram_svg` all shipped with this exact anti-pattern for a long time despite the rule being documented — check every SVG-returning builder in `pdf_context.py` for a stray `width="..." height="..."` on the root before assuming a sizing bug is CSS-only.
- **Every SVG chart's CSS rule must pair `width: 100%` with an explicit `max-width` equal to its builder's native viewBox width** — `width:100%` alone lets the chart stretch to fill whatever container it ends up in, which silently blows up bar height, font size and point size proportionally the moment the chart lands in a wider card than it was designed for (confirmed repeatedly: `.sector-bar-chart`, `.stacked-bar`, `svg.mini-bar-chart`, `svg.cvss-histogram`, `svg.epss-scatter` all had this gap and all looked "too big" as a result, several rounds in a row before the actual mechanism was identified). Multi-chart breakdowns belong in `.breakdown-col` (`.breakdown-grid`, ~320px per column) — never give one of them a lone `flex-basis: 100%` override to make it "the odd one out" full-width, unless the chart is genuinely meant to be full-width by design (map, history-line-chart, area-chart, spotlight card). A chart sitting in a narrower/wider container than its sibling charts of the same visual family is the #1 sign this rule was skipped.
- **The margin-over-gap pattern (above) leaves a dangling trailing margin on the last card of every wrapped row, not just the true last child** — a `flex-grow` row's items grow to fill `container width - total margins`, so the row's visible content always stops short by exactly one child's right-margin, on every row, not only the final one. Confirmed empirically as the root cause of "cards don't reach the full width of the card" across MTTK/C2/ThreatFox/Malicious URLs/Breaches. Fix: give the **parent** (`.kpi-grid`, `.chart-row`, `.breakdown-grid`) a `margin-right: -Npx` equal to its children's `margin-right`, cancelling the dangling edge on every row at once. `.chart-card--full`'s own right margin must match `.chart-card`'s (both 20px, bumped from 16px for more breathing room between chart cards) for `.chart-row`'s negative margin to compensate both variants correctly — don't let a "full-width" card variant use a different margin value than its siblings.
- **Jinja2 autoescape is ON (`pdf_renderer.py`, `select_autoescape(["html.j2"])`)** — any text field sourced from an external API (HIBP `description`, etc.) must be passed through `pdf_context.py::_strip_html()` before use, never inserted raw. Root cause of a real bug: HIBP's `Description` field routinely embeds raw `<a href=...>` markup; the old char-count truncation (`_BREACH_DESC_TRUNCATE_LEN`) could slice a tag in half, leaving an unclosed `<a>` that (with autoescape off, as it was) rendered as real HTML and swallowed every subsequent element in the document into one giant underlined link. `_strip_html()` must run **before** truncation, always — truncating raw HTML by character count is never safe regardless of autoescape state. The only HTML/SVG allowed to skip escaping is Python-generated markup (charts, markers), explicitly marked `| safe` in templates — audit that list before ever disabling autoescape again.
- **Links belong in dedicated elements (`macros.html.j2::source_link()`), never inline inside a free-text paragraph sourced from external data** — same reasoning as the autoescape point above: an inline `<a>` mixed into arbitrary external text is exactly the shape of bug that leaked formatting across the whole report once. New "read more" / source links go in their own `<span>`/`<p>` (`.source-link` class), never concatenated into a description string.
- **ransomware.live's documented API 404s; the real data lives at static dump URLs.** `api.ransomware.live/v2/*` (per the project's own docs) returns 404 on every endpoint tried. The actual working endpoints — `data.ransomware.live/posts.json` and `data.ransomware.live/groups.json`, full static dumps refreshed periodically — were found by inspecting the CSP headers on ransomware.live's own web pages, not by trusting the documentation. `config.py`'s `ransomware_live_posts_url`/`ransomware_live_groups_url` point at these; don't "fix" them back to the documented API without re-verifying it actually works first.
- **`posts.json` group names don't always match `groups.json` group names by strict equality** — casing/whitespace differences (confirmed empirically: several active groups failed a naive dict lookup). `extract/ransomware_live.py::normalize_group_name()` (lowercase + strip) is required on both sides of the join in `transform/ransomware.py::build_group_aggregates()`; don't join on the raw `name` field directly.
- **Filter-before-validate inversion for `ransomware_live.py`, deliberately opposite of `hibp.py`/`kev.py`'s convention** — `posts.json` is ~20MB/~30k rows; validating the full dump with Pydantic to then discard all but ~160 rows (a single week's window) is wasted work that scales with the dump's growth. `extract/ransomware_live.py::filter_posts_by_window()`/`filter_posts_last_n_weeks()` filter the raw dicts by date **before** `validate_batch()` ever sees them — the opposite order from every other extractor in this repo. This is intentional, not an oversight; don't "fix" it to match the usual pattern.
- **Ethics guarantee lives in the Pydantic schema, not just the template.** `validate/schemas.py::RansomwarePost` has no `post_title`/`website`/`post_url` fields at all — the raw-dict mappers in `extract/ransomware_live.py` drop them before validation even runs, so a victim name or leak-site URL cannot survive `model_validate()` even if a future edit accidentally left one in the raw dict. This is stronger than template-level discipline (autoescape, `_strip_html()`) alone: verified by `tests/test_extract_ransomware_live.py::test_ransomware_post_model_drops_victim_fields_even_if_present_in_raw_dict`. Any future field added to `RansomwarePost` must be checked against this guarantee before merging.

## Sources today

NVD (CVE), CISA KEV, abuse.ch FeodoTracker (C2 IPs)/URLhaus (malicious URLs)/ThreatFox (malware families), ip-api.com (geoloc), HaveIBeenPwned (breaches, primary) + BreachDirectory via RapidAPI (breaches, secondary cross-check, needs `RAPIDAPI_KEY` — degrades to a skipped count of 0 if absent, never blocks the run), FIRST.org EPSS (exploit probability score, merged onto CVE/KEV), ransomware.live (Ransomware Watch — own section, see exception above). Enrichment sources (Shodan InternetDB, Spamhaus DROP/EDROP, GreyNoise Community, OpenPhish) enrich these same entities per the "Data source integration rule" above — never add a source as an isolated new report section without checking that rule first. MITRE ATT&CK technique IDs are not a live source: `load/mitre_attack_map.py` is a small hand-curated dict (mirrors `cwe_names.py`'s pattern), not a fetch of the 47MB official STIX bundle.

**GreyNoise:** needs `GREYNOISE_API_KEY` forwarded into the job env in `.github/workflows/cron-etl.yml` (`env: GREYNOISE_API_KEY: ${{ secrets.GREYNOISE_API_KEY }}`) — the Python side (`config.py`) already reads the right env var name; a missing/empty key almost always means the workflow's `env:` block is missing this line, not a code or GitHub-secrets-store problem. Check that file first before touching `orchestrate.py`/`extract/greynoise.py`.

**`Illegal header value b'...\n'` on any API-key extractor:** not a missing/wrong env var — `config.py::_get_secret()` already `.strip()`s every API key read from env (`NVD_API_KEY`/`THREATFOX_AUTH_KEY`/`GREYNOISE_API_KEY`/`RAPIDAPI_KEY`) specifically because a GitHub Actions secret pasted with a trailing newline becomes an invalid HTTP header value the moment an extractor puts it straight into a `headers={}` dict. If this error reappears, check whether a *new* secret/extractor bypassed `_get_secret()` (e.g. read via `os.environ.get()` directly) rather than assuming the secret itself is broken.

**OpenPhish 302 redirect :** the shared `httpx.AsyncClient` in `orchestrate.py::run()` (one instance for every extraction source) must keep `follow_redirects=True` — `openphish.com/feed.txt` responds with a 302 to GitHub; without this flag, the HTML redirect page was ingested as if it were the text feed itself (historical root cause of broken OpenPhish parsing).

**ETL instrumentation (`source_item_counts`, `step_durations`) :** `orchestrate.py::run()` collects `source_item_counts` (dict source→item count, e.g. `{"nvd": 12, "openphish": 340, "hibp": 4}`) and `step_durations` (dict `extract`/`transform`/`load`→seconds via `time.monotonic()` checkpoints), added to `run_entry` in `finalize_manifest_and_index()`. Like `sources_status`, both fields live only in `runs/index.json` on S3 — never persisted to the `threat_report_runs` DB table (no DB migration). `GET /admin/threat-report/history` (backend) passes these through directly from the S3 entry.

## Testing

`pytest` — `tests/test_load_pdf_renderer.py` asserts on rendered CSS/HTML behavior directly (skipped if Pango/GTK isn't available, e.g. plain Windows without the WeasyPrint system deps). Extend this file rather than adding a parallel one when testing new template/CSS behavior.

**`OSError: cannot load library 'gobject-2.0-0'` on plain Windows :** not a code bug — WeasyPrint needs the GTK3 runtime (Pango/GObject/cairo) that Windows doesn't ship. CI (`.github/workflows/cron-etl.yml`, `runs-on: ubuntu-latest`) installs its own deps via `apt-get` and is unaffected. To render PDFs locally on Windows, install the [GTK3 runtime for Windows](https://github.com/tschoonj/GTK-for-Windows-Runtime-Environment-Installer) (`GTK3-Runtime-Win64` installer) and restart the shell so the DLLs are on `PATH`. Until then, `tests/test_load_pdf_renderer.py` self-skips (see above) rather than failing — this is expected on a fresh Windows machine, not a regression.
